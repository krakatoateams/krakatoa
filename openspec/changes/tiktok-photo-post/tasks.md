### 6b. Single mode (revised)
- [x] 6b.1 `UploadCard`'s "Upload Video" tab renamed "Upload Media"; its file filter/validation extends `ACCEPTED_VIDEO_TYPES` with an `ACCEPTED_IMAGE_TYPES` list (JPEG/PNG/WebP) so the same dropzone accepts both.
- [x] 6b.2 On a file landing (drop, file-picker, or an asset selected from the existing single-select video-history tab), derive `VideoItem.contentType` from that file's actual kind — video MIME → `"video"`, image MIME → `"photo"`. No user-facing control sets this directly anymore.
- [x] 6b.3 New raw-photo upload path (design.md Decision 8): `/api/upload/sign` extended with an optional `mediaType` field (defaults to `"video"` for full backward compat) — `"image"` validates against a JPEG/PNG/WebP allow-list + 10MB ceiling (matching `api/generate-photo`) and signs into `photosStoragePath("scheduler-uploads", ...)` instead of `videosStoragePath`, so the result already satisfies the task-3 proxy's `photos/`-only check with no proxy changes needed. Resolved per Decision 8: stays ephemeral/schedule-only (no `user_creations` row), matching video's existing precedent.
- [x] 6b.4 Once an item's `contentType` is `"photo"` (from a raw drop or from picking one existing photo), the multi-select `CreationsHistory` picker (kept from the first pass, task 6.2) is still available on that card so the user can add more existing photos toward the 35-photo cap — dropping one photo doesn't replace the "add more from history" option, it's just no longer the only entry point.
- [x] 6b.5 `PlatformFields` gains a `contentType` prop: when `"photo"`, the YouTube checkbox renders disabled/grayed with a tooltip ("YouTube doesn't support photo posts") instead of being simply hidden or silently reset later. Replaces the old toggle's `useEffect`-based reset-on-conflict (removed from `ScheduleCard` — the invalid combination is unselectable at the source).
- [x] 6b.6 Submit logic: sends `photo_urls` instead of `video_url` for the TikTok request when the item's `contentType` is `"photo"` — same as before, just driven by the derived field instead of the removed manual toggle.
- [x] 6b.7 Edge case (design.md Risks): a photo-typed card whose photos are all removed via the picker can't be scheduled (same as today's empty-draft guard, `ScheduleCard.isReady`/`hasMedia`) rather than silently reverting to video/re-enabling YouTube.
